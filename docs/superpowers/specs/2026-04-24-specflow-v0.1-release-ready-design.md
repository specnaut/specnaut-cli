# Specflow v0.1 release-ready bundle — Design

**Date** : 2026-04-24 **Statut** : draft, en attente de revue **Scope** : 3 sous-briques packagées
ensemble pour rendre Specflow publiquement distribuable **Prérequis** : v0.1-init + backlog-sync
mergés sur `main` (126 tests green)

---

## 1. Objectifs et non-objectifs

### Objectifs

Livrer trois additions courtes qui, ensemble, transforment Specflow d'un "ça marche sur la machine
de Kevin" en "installable et débuggable par n'importe qui" :

1. **`specflow check`** — diagnostic de l'environnement et d'une install Specflow existante
   (prerequisites + project health).
2. **`--force` sur `specflow init`** — permettre de re-scaffolder un projet avec `.claude/` ou
   `.specify/` déjà présent, en sauvegardant les fichiers existants avant overwrite.
3. **`install.sh` + Homebrew formula** — deux canaux d'installation reproductibles pour le binaire.

### Non-objectifs

- `specflow upgrade` (diff template 3-way merge) — reporté en brique suivante.
- Auto-update après `self-update` succès (reboot/relaunch) — pas demandé.
- Multi-tap Homebrew (core, official) — on fournit uniquement un tap personnel
  `kevinraimbaud/homebrew-tap`.
- Signature/notarization des binaires (macOS Gatekeeper) — reporté en v0.2+.

## 2. Sous-brique 1 — `specflow check`

### Surface CLI

```
specflow check [--project]
```

Sans argument : vérifie l'**environnement** (binaires externes + versions). Avec `--project` : en
plus, vérifie l'**install Specflow** du cwd (fichiers présents, config valide, version templates
cohérente).

### Sortie

Table ASCII compacte, exit 0 si tout OK, exit 1 sinon :

```
specflow check — environment

  git         ✓ git version 2.43.0
  gh          ✓ gh version 2.50.0 (authenticated as kevinraimbaud)
  deno        ⚠ deno not in PATH (optional — only needed for self-update on non-compiled runs)

specflow check — project (./)

  .specify/               ✓ present
  .claude/                ✓ present
  constitution            ✓ .specify/memory/constitution.md (not empty)
  backlog config          ✓ .specflow/config.yml (github:kevinraimbaud/specflow → project #3)
  templates version       ✓ matches binary (0.2.0)

All checks passed.
```

### Architecture (DDD continuée)

- **Domain** : `src/domain/check_result.ts` — `CheckResult`, `CheckOutcome` value types (pure).
- **Application** : `src/application/run_checks.ts` — `RunChecksUseCase` (orchestrates N independent
  checks through ports).
- **Application ports additions** (`ports.ts`) :
  - `EnvironmentProbe` — lookups for `git`, `gh`, `deno` versions + `gh auth status`.
  - `ProjectInspector` — checks file presence, reads constitution emptiness, reads `config.yml`,
    compares `TEMPLATES_VERSION`.
- **Infrastructure** :
  - `DenoEnvironmentProbe` — shells out via `SubprocessRunner` (already exists).
  - `FsProjectInspector` — reads filesystem via `Deno.stat`/`readTextFile`.
- **CLI handler** : `src/cli/handlers/check_handler.ts` — renders the table.

### Tests

- `tests/application/run_checks_test.ts` — 4 tests (all pass / mixed / project mode / project mode
  with missing files). Inject fake probes.
- `tests/infrastructure/fs_project_inspector_test.ts` — 3 tests against tmp dirs (well-formed
  project, missing .specify, old templates version).

## 3. Sous-brique 2 — `specflow init --force`

### Surface CLI

Extension du parser existant : ajouter `--force` au flag boolean set de l'`init` intent. Déjà
réservé dans la spec v0.1 (§6 de la design initiale), maintenant implémenté.

### Sémantique

1. Sans `--force` : comportement actuel (abandon si conflit).
2. Avec `--force` :
   - Pour chaque fichier managé qui existe déjà, le renommer `{path}.specflow.bak` (suffixe fixe —
     pas de timestamp pour rester déterministe). Si `{path}.specflow.bak` existe déjà d'un run
     précédent, l'écraser (la règle "on ne perd rien" est assumée par le nom explicite
     `.specflow.bak`, charge à l'utilisateur de préserver ses modifs).
   - Puis écrire le template par-dessus comme d'habitude.
   - Si un `.specflow.bak` est écrit, le logger :
     `↳ backed up existing .claude/commands/speckit.specify.md → .claude/commands/speckit.specify.md.specflow.bak`.
3. Les fichiers **non managés** par Specflow (hors du manifest) restent intacts, même avec
   `--force`.

### Architecture

- **Port existant modifié** : `FsWriter.writeBundle(bundle, targetDir, { overwrite: true })` signale
  déjà qu'il peut écraser. Il faut ajouter une option `{ backupExisting: true }` qui fait le rename
  avant chaque écriture et retourne la liste des backups effectués.
- **Domain unchanged**.
- **CLI parser** : ajouter `"force"` au boolean set dans parser.ts, propager dans l'`InitIntent`.
- **Handler init** : si `intent.force`, passer `backupExisting: true`.
- **`InitProjectUseCase`** : si `force=true`, skip la `detectConflicts` gate (puisqu'on accepte
  l'overwrite) et passe `{ overwrite: true, backupExisting: true }` au writer. Expose
  `backups: string[]` dans `InitResult` pour que le handler les affiche.

### Tests

- Unit `DenoFsWriter` : 2 tests (backup creates .specflow.bak; existing .specflow.bak is
  overwritten).
- Integration `specflow init --force` in tmp dir with pre-existing `.claude/` — exit 0,
  `.specflow.bak` files present, original files replaced.

## 4. Sous-brique 3 — `install.sh` + Homebrew formula

### 4.1 `install.sh`

Script `curl | sh`-friendly. Détecte OS/arch, télécharge le bon binaire depuis la dernière release,
vérifie SHA256, installe dans `$PREFIX` (default `/usr/local/bin`).

Usage :

```bash
curl -fsSL https://raw.githubusercontent.com/kevinraimbaud/specflow/main/install.sh | bash
# or: pin a version
curl -fsSL https://raw.githubusercontent.com/kevinraimbaud/specflow/main/install.sh | VERSION=v0.1.0-alpha.1 bash
# or: install to a different prefix
curl -fsSL https://raw.githubusercontent.com/kevinraimbaud/specflow/main/install.sh | PREFIX=$HOME/.local/bin bash
```

Comportement :

1. Détecte `uname -s` + `uname -m` → maps vers `specflow-<os>-<arch>[.exe]`.
2. Si `VERSION` non fourni, query
   `https://api.github.com/repos/kevinraimbaud/specflow/releases/latest`.
3. Download `specflow-<triple>` + `specflow-<triple>.sha256` via `curl -fsSL`.
4. Verify `shasum -a 256 -c` (or `sha256sum` on Linux).
5. Chmod 755 + move vers `$PREFIX/specflow`. Si `$PREFIX` requires sudo, prompt (but don't
   auto-sudo).
6. Print `specflow --version` for confirmation.

Robustesse :

- `set -euo pipefail` au top.
- Erreurs explicites si curl/shasum manquent.
- OS/arch non-supporté → liste des cibles connues.
- Windows : refuse avec instruction de télécharger manuellement depuis GH Releases (Windows shell
  script dehors du scope).

### 4.2 Homebrew formula

Tap repository séparé : `kevinraimbaud/homebrew-tap`. Formula file : `Formula/specflow.rb`.

```ruby
class Specflow < Formula
  desc "Spec-driven CLI with auto-chain, review phase, and backlog sync"
  homepage "https://github.com/kevinraimbaud/specflow"
  version "0.1.0-alpha.1"

  on_macos do
    on_arm do
      url "https://github.com/kevinraimbaud/specflow/releases/download/v#{version}/specflow-macos-arm64"
      sha256 "..." # populated at release
    end
    on_intel do
      url "https://github.com/kevinraimbaud/specflow/releases/download/v#{version}/specflow-macos-x64"
      sha256 "..."
    end
  end

  on_linux do
    on_arm do
      url "https://github.com/kevinraimbaud/specflow/releases/download/v#{version}/specflow-linux-arm64"
      sha256 "..."
    end
    on_intel do
      url "https://github.com/kevinraimbaud/specflow/releases/download/v#{version}/specflow-linux-x64"
      sha256 "..."
    end
  end

  def install
    bin.install Dir["*"].first => "specflow"
  end

  test do
    assert_match "specflow #{version}", shell_output("#{bin}/specflow --version")
  end
end
```

**Automation**: extend `.github/workflows/release.yml` to push the formula file to
`kevinraimbaud/homebrew-tap` after a successful release. v0.1 de cette brique = **formula file
committed dans le repo Specflow** (`packaging/homebrew/specflow.rb` template, `.sha256` values
placeholders updated by the release workflow). **L'auto-publish vers le tap est reporté** — on
décrit juste comment le faire manuellement.

### Tests

- `install.sh` → shellcheck clean + `install.sh --dry-run` option qui skip le download/move et print
  ce qu'il FERAIT (pour tests).
- Homebrew formula : `brew audit --strict --online` (skip en CI, doc uniquement).

## 5. Impact sur le binaire et les permissions

- `specflow check` nécessite `--allow-run=git,gh,deno` et `--allow-read`. Déjà baked.
- `specflow init --force` : pas de nouvelle permission (juste rename + write via
  Deno.rename/writeFile, déjà OK).
- `install.sh` : bash pur, pas de Deno. Indépendant du binaire.
- Homebrew formula : pas de Deno non plus.

Aucun changement à `scripts/build.ts` ou `.github/workflows/release.yml` requis pour les
permissions. La release workflow pourrait être étendue pour aussi uploader `install.sh` à côté des
binaires et committer dans le tap, mais c'est une extension optionnelle.

## 7. Risques et points ouverts

- **`install.sh` signature** : pas de notarization macOS en v0.1. L'utilisateur devra
  `xattr -d com.apple.quarantine` ou passer par System Settings à la première exécution. À
  documenter clairement dans le script + README.
- **Homebrew auto-publish** : formellement reporté. En attendant, release process = 4 steps manuels
  (voir `.claude/skills/release/SKILL.md` existant, à étendre pour inclure le tap).
- **`specflow check --project` stricteness** : définir si une absence de `.specflow/config.yml` est
  ⚠ warn ou ❌ fail. Décision : ⚠ warn (config is optional until user wants sync).
- **`--force` edge case** : que faire si `.specflow.bak` est lui-même tracké par git et committé ?
  Décision : le `.gitignore` template embarque déjà dist/ etc. ; ajouter `*.specflow.bak` au
  `.gitignore` bundled via les templates root.
