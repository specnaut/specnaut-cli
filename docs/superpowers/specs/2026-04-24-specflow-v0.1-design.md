# Specflow v0.1 — Design

**Date** : 2026-04-24 **Status** : draft, pending review **Language** : Deno (TypeScript) **Target
users** : Kevin (author) first, then "Spec Kit + Claude Code" community next

---

## 1. Goals and non-goals

### Goals v0.1

1. Provide a **native cross-platform binary** (macOS arm64/x64, Linux x64/arm64, Windows x64) that
   replaces upstream's `specify init` on the end user's side — no more Python, no more `uv`.
2. Scaffold a project ready to consume via **Claude Code only** (`.claude/` + `.specify/` in the
   right format).
3. Bundle by default the **3 differentiators** vs upstream:
   - auto mode (`specify` chains until merge, stops only at clarifications and before merge);
   - structured `review` phase (architecture + quality gates + implement→review→fix loop);
   - `backlog` command + Product Owner agent + one-way GitHub Issues/Project sync.
4. Be **distributable** via personal Homebrew tap, `curl | sh`, and GitHub Releases.

### Non-goals v0.1

- Support for any other AI harness (Cursor, Codex, Copilot, Gemini, Windsurf, Qwen, Tabnine…) —
  deferred to v0.2+.
- Bidirectional sync with GitHub (v0.1 = push MD → GH only; reverse sync = v0.2).
- GitLab/Bitbucket integration — v0.3+.
- Containerization, devcontainers, VS Code extensions — no.
- Talking to an LLM directly. Specflow makes **no API calls**.
- Agent runtime / autonomous orchestrator — explicitly out of scope (Claude Code orchestrates).

## 2. Functional scope v0.1

### CLI Surface

| Command                                                                      | Description                                                                                                                                                                                                           |
| ---------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `specflow init [<project-name>] [--here] [--ai claude] [--no-git] [--force]` | Bootstrap a project: creates `.claude/` + `.specify/` + `tasks/` from embedded templates. `--here` = in current cwd. `--ai` fixed to `claude` in v0.1. `--force` = overwrites existing `.claude/` after confirmation. |
| `specflow check`                                                             | Verifies the environment (git, gh, node/deno if the project needs it) and the consistency of an existing Specflow install.                                                                                            |
| `specflow upgrade`                                                           | Updates generated files to the latest version of embedded templates, preserving user customizations where feasible (strategy to be defined — v0.1 = diff + prompt).                                                   |
| `specflow --version`                                                         | Shows the binary version + the version of embedded templates.                                                                                                                                                         |
| `specflow --help`                                                            | Help.                                                                                                                                                                                                                 |

All other surfaces (`specify`, `clarify`, `plan`, `tasks`, `analyze`, `implement`, `review`,
`merge`, `backlog`, `constitution`, `checklist`) **are not Specflow commands** — they are Claude
Code commands generated in `.claude/commands/*.md` and executed by Claude Code.

### Behavior of `specflow init`

1. Determines the target directory (cwd if `--here`, otherwise creates `./<project-name>`).
2. Detects if a git project already exists. If not, offers `git init` (unless `--no-git`).
3. Detects an existing `.claude/` or `.specify/`. If present without `--force`, aborts with a clear
   message. If `--force`, merges (see §6).
4. Writes files from embedded templates:
   - `.specify/memory/constitution.md` (empty with project placeholder)
   - `.specify/templates/{spec,plan,tasks,checklist,constitution,agent-file}-template.md`
   - `.specify/scripts/bash/{check-prerequisites,common,create-new-feature,setup-plan,update-agent-context}.sh`
   - `.claude/commands/speckit.{specify,clarify,plan,tasks,analyze,implement,review,merge,constitution,checklist}.md`
   - `.claude/commands/backlog.md`
   - `.claude/agents/{product-owner,developer,review-coordinator,code-reviewer,security-auditor,test-reviewer,qa-tester,workflow-manager}.md`
   - `.claude/skills/speckit/SKILL.md` (auto-chain dispatcher)
   - `.claude/skills/backlog/scripts/sync-to-github.py` (sync script, Python acceptable here because
     executed by end user via `gh` CLI)
   - `tasks/backlog.md` (empty with header)
   - `CLAUDE.md` at root (pointer to `.claude/`)
   - `AGENTS.md` placeholder
5. If `gh` available: offers to create a GitHub Project V2 for the backlog. Optional.
6. Displays a summary and first suggested step (`/backlog add "first task"`).

### Internal Technical Choices

- **CLI framework** : `@std/cli/parse-args` is enough for v0.1 scope (5 commands, some flags). We
  avoid a Cliffy dependency to stay zero-deps; we re-evaluate if TUI needs grow.
- **TUI** : `@std/cli/unstable-spinner` + `@std/fmt/colors`. No heavy TUI framework.
- **YAML** (frontmatter in templates) : `@std/yaml`.
- **Markdown frontmatter** : in-house parsing (3 lines) or `@std/front-matter`.
- **Embedded templates** : each template file is imported via `import … with { type: "text" }` (if
  stabilized) or, by default in v0.1, **inlined as TypeScript constant strings** generated by a
  `scripts/bundle-templates.ts` script at pre-build. This ensures that after `deno compile`, the
  binary embeds **all** templates without relying on the filesystem.

## 4. The 3 Differentiators — Design

### 4.1 Auto mode (auto-chain)

**Where it lives** : in `.claude/skills/speckit/SKILL.md` (the same file as
`examples/.claude/skills/speckit/SKILL.md`, reused and generalized).

**Mechanism** : Claude Code loops, not Specflow. The SKILL.md explicitly describes to Claude Code to
invoke the next command via the `Skill` tool after each phase. User stops are limited to two:

- STOP #1 : spec clarifications if `[NEEDS CLARIFICATION]` markers are unresolved.
- STOP #2 : human validation before merge.

**Opt-out** : `/speckit.specify --manual "<feat>"` → single phase at a time, upstream behavior.

**Cleanup vs examples** :

- Remove all `--copilot` / Copilot Coding Agent handoff references (out of scope v0.1).
- Remove mentions of `configcat`, `pulumi`, etc. project-specific terms.
- Final SKILL.md is 100% agnostic of the consuming project's stack.

### 4.2 Structured `review` Phase

**Where it lives** : in `.claude/commands/speckit.review.md` + the `review-coordinator.md` agent +
sub-reviewers `code-reviewer.md`, `security-auditor.md`, `test-reviewer.md`.

**Mechanism** : Claude Code command that:

1. Identifies modified files (`git diff --name-only <base>`).
2. Runs language-agnostic static checks:
   - Silent catches (pattern `catch \(.*\) \{\s*\}`) — universal.
   - Exposure of internal IDs in routes / URLs (generic heuristic).
   - New TODO / FIXME.
3. Delegates to sub-agents for contextual review (architecture, security, tests).
4. Runs auto-detected **quality gates** :
   - `package.json` present → `npm run format` / `lint` / `typecheck` / `test` if scripts exist.
   - `deno.json` present → `deno fmt` / `lint` / `check` / `test`.
   - `Cargo.toml` → `cargo fmt` / `clippy` / `test`.
   - `go.mod` → `go fmt` / `vet` / `test`.
   - `pyproject.toml` → `ruff` / `mypy` / `pytest` if declared.
5. If CRITICAL/HIGH : loop → `@developer` fixes → targeted re-review.

**Cleanup vs examples** :

- Remove all AdonisJS / Lucid / Bouncer / VineJS / Inertia specific checks.
- Keep generic principles (thin controllers, no silent catches, DI, layer separation) **as questions
  to ask the LLM**, not as hard-coded rules.
- The "project constitution" (`.specify/memory/constitution.md`) is where the user declares their
  own invariants; the reviewer reads and applies them.

### 4.3 `backlog` Command + Product Owner Agent

**Generated files** :

- `.claude/commands/backlog.md` — dispatcher of sub-commands (`list`, `next`, `add`, `update`,
  `groom`, `brief`, `sync`, …).
- `.claude/agents/product-owner.md` — agent with `memory: project`, exclusive responsibility for
  mutations.
- `.claude/skills/backlog/scripts/sync-to-github.py` — Python script (allowed here because executed
  by end user via `gh`, not by Specflow). **Note** : we can rewrite this script in Deno (`.ts`
  called via `deno run`) to avoid Python dependency on user's end, **and this is recommended in
  v0.1** to keep the promise "no dependencies".
- `tasks/backlog.md` — empty index.
- `tasks/backlog/` — empty folder.

**v0.1 Decision** : rewrite `sync-to-github.ts` in Deno, invoked via
`deno run -A sync-to-github.ts …`. This implies that **Deno remains available on user's end** —
partial contradiction with the "binary only" promise. Alternative: include sync in the Specflow
binary itself (`specflow backlog sync …`). This alternative is better: no Deno dependency on user's
end for sync.

**Resolution** : the `specflow` binary exposes **one** additional sub-command
`specflow backlog sync` that reads `tasks/backlog.md`, calls `gh` to create/update issues, manages
Project V2 updates. The PO agent generates in its responses the directive
`Run: specflow backlog sync` that the orchestrator executes. **This adds `backlog sync` to the CLI
surface**, alongside `init`/`check`/`upgrade`.

**Task frontmatter schema** :

```yaml
---
id: NNN                # zero-padded 3 digits
title: string
category: string       # free, suggested by PO
priority: string       # critical | high | medium | low
complexity: number     # Fibonacci 1,2,3,5,8,13,21
status: string         # todo | in_progress | done | deferred | blocked
depends_on: [string]
spec: string | null    # id of SpecKit spec if attached
tags: [string]
created: YYYY-MM-DD
---
```

Identical to the example. Generic.

## 6. Management of Existing Installs (`--force`, `upgrade`)

**Case 1 — blank folder** : no `.claude/` or `.specify/` → direct write.

**Case 2 — `.claude/` exists (user already has Claude Code config)** :

- Without `--force` → abort with error "present — use `specflow upgrade` or `--force`".
- With `--force` :
  - Files Specflow manages (declarative list in `templates/manifest.json`) → overwrite.
  - Unknown user files → left intact.
  - Known conflicts (e.g., user has custom `speckit.implement.md`) → backup to `.specflow.bak` and
    replace.

**`upgrade`** : walks the manifest, compares checksums, lists diffs, prompts for each change
(`overwrite / keep / show diff`). Manifest version = templates version.

## 7. Distribution

### `deno compile` Targets

```
deno compile --target x86_64-unknown-linux-gnu  -o dist/specflow-linux-x64    src/main.ts
deno compile --target aarch64-unknown-linux-gnu -o dist/specflow-linux-arm64  src/main.ts
deno compile --target x86_64-apple-darwin       -o dist/specflow-macos-x64    src/main.ts
deno compile --target aarch64-apple-darwin      -o dist/specflow-macos-arm64  src/main.ts
deno compile --target x86_64-pc-windows-msvc    -o dist/specflow-windows-x64.exe src/main.ts
```

### Channels

- **GitHub Releases** — tags `vX.Y.Z`, release.yml workflow triggered on tag. Upload 5 binaries +
  SHA256 checksums.
- **Personal Homebrew tap** (`kevinr/homebrew-tap`) — Ruby formula pointing to the release.
  Auto-update on each release via PR.
- **`curl | sh` installer** — `install.sh` script on GitHub Pages or in the repo
  (`https://specflow.dev/install.sh` if domain, otherwise raw github URL). Detects OS/arch,
  downloads the right binary.

### Versioning

- Strict semver.
- Binary version + embedded templates version displayed by `--version`. Ex :
  `specflow 0.1.0 (templates 0.1.0)`. They can diverge starting v0.2.

## 8. Error Handling and Edge Cases

| Case                                    | Behavior                                                                               |
| --------------------------------------- | -------------------------------------------------------------------------------------- |
| No `git` in PATH                        | `init` continues (we don't block), `check` warns. `backlog sync` refuses.              |
| No `gh` in PATH                         | `init` continues, `backlog sync` refuses with install instruction.                     |
| Network absent (install)                | Templates are inlined in the binary, so `init` works **offline**. No runtime download. |
| `.claude/` already exists               | See §6.                                                                                |
| Target directory does not exist         | Created recursively if `init <path>`.                                                  |
| Invalid project name (shell characters) | Rejected with message.                                                                 |
| Non-interactive terminal (CI)           | All interactions default to "no"; explicit flags required.                             |
| `gh` not authenticated                  | `backlog sync` detects via `gh auth status` and offers `gh auth login`.                |

## 9. Tests

- **Unit** : manifest parsing, template rendering, environment detection. Target ~70% line coverage.
- **Integration** : `deno test` that spawns `specflow init` in a `Deno.makeTempDir()`, verifies:
  - expected directory tree,
  - content of key files (checksums),
  - idempotence (`init` on blank folder × 2),
  - `--force` overwrite correct.
- **Smoke CI** : GH Actions matrix on 3 OS (ubuntu, macos, windows) to validate that `specflow init`
  produces the same files everywhere.

## 11. Risks and Open Questions

1. **Upstream Template Drift** — Spec Kit upstream evolves (they are at v0.8.1.dev). We need a
   monitoring process (periodic diff) to avoid accumulating too much debt. Plan a minimum quarterly
   rebase.
2. **License** — confirmed MIT (GitHub, Inc.). Explicit attribution required in `LICENSE` and
   headers of reused files.
3. **Binary Size** — `deno compile` produces ~80 MB. Manageable but should be announced clearly. No
   solution except migrating to Rust/Go later.
4. **Templates Embedding** — wait for stabilization of
   `import text from "./x.md" with { type: "text" }` in Deno, or use a pre-compile script that
   generates a `templates.ts` module with all content. Option 2 retained in v0.1 for zero dependency
   on non-stable features.
5. **Python in Sync Script** — resolved: we rewrite in TypeScript and put it **in the binary**
   (sub-command `specflow backlog sync`).
6. **Windows Support for Bash Scripts** — upstream has `scripts/powershell/` for Windows. Should
   bundle those too or replace with Deno scripts run via `deno run` (= Deno dependency on user end).
   Decision: **bundle both** bash + powershell versions (like upstream). No added Deno dependency on
   user end.
