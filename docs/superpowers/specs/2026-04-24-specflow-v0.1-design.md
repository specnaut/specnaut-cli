# Specflow v0.1 — Design

**Date** : 2026-04-24 **Statut** : draft, en attente de revue **Langage** : Deno (TypeScript)
**Cible utilisateur** : Kevin (auteur) d'abord, communauté "Spec Kit + Claude Code" ensuite

---

## 1. Objectifs et non-objectifs

### Objectifs v0.1

1. Fournir un **binaire natif cross-platform** (macOS arm64/x64, Linux x64/arm64, Windows x64) qui
   remplace `specify init` d'upstream côté utilisateur final — plus de Python, plus de `uv`.
2. Scaffolder un projet prêt à consommer via **Claude Code uniquement** (`.claude/` + `.specify/` au
   bon format).
3. Embarquer par défaut les **3 différenciateurs** vs upstream :
   - mode auto (`specify` enchaîne jusqu'au merge, s'arrête seulement aux clarifications et avant
     merge) ;
   - phase `review` structurée (architecture + quality gates + boucle implement→review→fix) ;
   - commande `backlog` + agent Product Owner + sync GitHub Issues/Project one-way.
4. Être **distribuable** via Homebrew tap personnel, `curl | sh` et GitHub Releases.

### Non-objectifs v0.1

- Support de tout autre harnais IA (Cursor, Codex, Copilot, Gemini, Windsurf, Qwen, Tabnine…) —
  reporté en v0.2+.
- Sync bidirectionnel avec GitHub (v0.1 = push MD → GH seulement ; reverse sync = v0.2).
- Intégration GitLab/Bitbucket — v0.3+.
- Conteneurisation, devcontainers, extensions VS Code — non.
- Parler à un LLM directement. Specflow ne fait **aucun appel API**.
- Agent runtime / orchestrateur autonome — explicitement hors scope (c'est Claude Code qui
  orchestre).

## 2. Scope fonctionnel v0.1

### Surface CLI

| Commande                                                                     | Description                                                                                                                                                                                                                 |
| ---------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `specflow init [<project-name>] [--here] [--ai claude] [--no-git] [--force]` | Bootstrap un projet : crée `.claude/` + `.specify/` + `tasks/` selon les templates embarqués. `--here` = dans le cwd courant. `--ai` fixé à `claude` en v0.1. `--force` = écrase un `.claude/` existant après confirmation. |
| `specflow check`                                                             | Vérifie l'environnement (git, gh, node/deno si le projet en a besoin) et la cohérence d'une install Specflow existante.                                                                                                     |
| `specflow upgrade`                                                           | Met à jour les fichiers générés vers la dernière version des templates embarqués, en préservant les customisations utilisateur là où c'est faisable (stratégie à définir — v0.1 = diff + prompt).                           |
| `specflow --version`                                                         | Affiche la version du binaire + la version des templates embarqués.                                                                                                                                                         |
| `specflow --help`                                                            | Aide.                                                                                                                                                                                                                       |

Toutes les autres surfaces (`specify`, `clarify`, `plan`, `tasks`, `analyze`, `implement`, `review`,
`merge`, `backlog`, `constitution`, `checklist`) **ne sont pas des commandes Specflow** — ce sont
des commandes Claude Code générées dans `.claude/commands/*.md` et exécutées par Claude Code.

### Comportement de `specflow init`

1. Détermine le répertoire cible (cwd si `--here`, sinon crée `./<project-name>`).
2. Détecte si un projet git existe déjà. Si non, propose `git init` (sauf `--no-git`).
3. Détecte un `.claude/` ou `.specify/` existant. Si présent sans `--force`, abandonne avec un
   message clair. Si `--force`, merge (voir §6).
4. Écrit les fichiers depuis les templates embarqués :
   - `.specify/memory/constitution.md` (vide avec placeholder projet)
   - `.specify/templates/{spec,plan,tasks,checklist,constitution,agent-file}-template.md`
   - `.specify/scripts/bash/{check-prerequisites,common,create-new-feature,setup-plan,update-agent-context}.sh`
   - `.claude/commands/speckit.{specify,clarify,plan,tasks,analyze,implement,review,merge,constitution,checklist}.md`
   - `.claude/commands/backlog.md`
   - `.claude/agents/{product-owner,developer,review-coordinator,code-reviewer,security-auditor,test-reviewer,qa-tester,workflow-manager}.md`
   - `.claude/skills/speckit/SKILL.md` (dispatcher auto-chain)
   - `.claude/skills/backlog/scripts/sync-to-github.py` (sync script, Python acceptable ici car
     exécuté par l'utilisateur final via `gh` CLI)
   - `tasks/backlog.md` (vide avec en-tête)
   - `CLAUDE.md` au root (pointeur vers `.claude/`)
   - `AGENTS.md` placeholder
5. Si `gh` disponible : propose de créer un GitHub Project V2 pour le backlog. Optionnel.
6. Affiche un résumé et le premier pas suggéré (`/backlog add "première tâche"`).

### Choix techniques internes

- **CLI framework** : `@std/cli/parse-args` suffit pour le scope v0.1 (5 commandes, qq flags). On
  évite une dépendance à Cliffy pour rester zero-deps ; on réévalue si le besoin TUI grandit.
- **TUI** : `@std/cli/unstable-spinner` + `@std/fmt/colors`. Pas de framework TUI lourd.
- **YAML** (frontmatter dans templates) : `@std/yaml`.
- **Markdown frontmatter** : parsing maison (3 lignes) ou `@std/front-matter`.
- **Templates embarqués** : chaque fichier template est importé via `import … with { type: "text" }`
  (si stabilisé) ou, par défaut v0.1, **inlined en tant que chaînes constantes TypeScript** générées
  par un script `scripts/bundle-templates.ts` au pré-build. Cela garantit qu'après `deno compile`,
  le binaire embarque **tous** les templates sans dépendre du filesystem.

## 4. Les 3 différenciateurs — design

### 4.1 Mode auto (auto-chain)

**Où vit-il** : dans `.claude/skills/speckit/SKILL.md` (le même fichier que celui de
`examples/.claude/skills/speckit/SKILL.md`, repris et généralisé).

**Mécanisme** : c'est Claude Code qui boucle, pas Specflow. Le SKILL.md décrit explicitement à
Claude Code d'invoquer la commande suivante via le `Skill` tool après chaque phase. Les stops
utilisateur sont limités à deux :

- STOP #1 : clarifications du spec si `[NEEDS CLARIFICATION]` markers non résolus.
- STOP #2 : validation humaine avant merge.

**Opt-out** : `/speckit.specify --manual "<feat>"` → une seule phase à la fois, comportement
upstream.

**Nettoyage vs exemples** :

- Retirer toute référence `--copilot` / handoff Copilot Coding Agent (hors scope v0.1).
- Retirer les mentions de `configcat`, `pulumi`, etc. projet-spécifiques.
- Le SKILL.md final est 100 % agnostique du stack du projet consommateur.

### 4.2 Phase `review` structurée

**Où vit-elle** : dans `.claude/commands/speckit.review.md` + l'agent `review-coordinator.md` + les
sous-reviewers `code-reviewer.md`, `security-auditor.md`, `test-reviewer.md`.

**Mécanisme** : commande Claude Code qui :

1. Identifie les fichiers modifiés (`git diff --name-only <base>`).
2. Lance des checks statiques **agnostiques du langage cible** :
   - Silent catches (pattern `catch \(.*\) \{\s*\}`) — universel.
   - Exposition d'IDs internes dans routes / URLs (heuristique générique).
   - TODO / FIXME nouveaux.
3. Délègue à des sous-agents pour la relecture contextuelle (architecture, sécurité, tests).
4. Lance les **quality gates** auto-détectées :
   - `package.json` présent → `npm run format` / `lint` / `typecheck` / `test` si les scripts
     existent.
   - `deno.json` présent → `deno fmt` / `lint` / `check` / `test`.
   - `Cargo.toml` → `cargo fmt` / `clippy` / `test`.
   - `go.mod` → `go fmt` / `vet` / `test`.
   - `pyproject.toml` → `ruff` / `mypy` / `pytest` si déclarés.
5. Si CRITICAL/HIGH : boucle → `@developer` fixe → re-review ciblée.

**Nettoyage vs exemples** :

- Supprimer tous les checks AdonisJS / Lucid / Bouncer / VineJS / Inertia spécifiques.
- Garder les principes génériques (thin controllers, no silent catches, DI, layer separation) **sous
  forme de questions à poser au LLM**, pas comme règles hard-codées.
- La "constitution projet" (`.specify/memory/constitution.md`) est l'endroit où l'utilisateur
  déclare ses propres invariants ; le reviewer les lit et les applique.

### 4.3 Commande `backlog` + agent Product Owner

**Fichiers générés** :

- `.claude/commands/backlog.md` — dispatcher de sous-commandes (`list`, `next`, `add`, `update`,
  `groom`, `brief`, `sync`, …).
- `.claude/agents/product-owner.md` — agent avec `memory: project`, responsable exclusif des
  mutations.
- `.claude/skills/backlog/scripts/sync-to-github.py` — script Python (autorisé ici car exécuté par
  l'utilisateur final via `gh`, pas par Specflow). **Note** : on peut réécrire ce script en Deno
  (`.ts` appelé via `deno run`) pour éviter la dépendance Python côté utilisateur, **et c'est
  recommandé en v0.1** pour tenir la promesse "aucune dépendance runtime".
- `tasks/backlog.md` — index vide.
- `tasks/backlog/` — dossier vide.

**Décision v0.1** : réécrire le `sync-to-github.ts` en Deno, invoqué via
`deno run -A sync-to-github.ts …`. Cela implique que **Deno reste disponible côté utilisateur** —
contradiction partielle avec la promesse "binaire seul". Alternative : inclure le sync dans le
binaire Specflow lui-même (`specflow backlog sync …`). Cette alternative est meilleure : aucune
dépendance Deno côté utilisateur pour le sync.

**Résolution** : le binaire `specflow` expose **une** sous-commande supplémentaire
`specflow backlog sync` qui lit `tasks/backlog.md`, appelle `gh` pour créer/mettre à jour les
issues, gère la mise à jour du Project V2. L'agent PO génère dans ses réponses le directive
`Run: specflow backlog sync` que l'orchestrateur exécute. **Ceci ajoute `backlog sync` à la surface
CLI**, à côté de `init`/`check`/`upgrade`.

**Schéma frontmatter des tâches** :

```yaml
---
id: NNN                # zero-padded 3 digits
title: string
category: string       # libre, suggéré par le PO
priority: string       # critical | high | medium | low
complexity: number     # Fibonacci 1,2,3,5,8,13,21
status: string         # todo | in_progress | done | deferred | blocked
depends_on: [string]
spec: string | null    # id de spec SpecKit si attachée
tags: [string]
created: YYYY-MM-DD
---
```

Identique à l'exemple. Générique.

| Source upstream                                                                                             | Décision                                                                         |
| ----------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------- |
| `templates/spec-template.md`                                                                                | **Réutiliser tel quel** (agnostique)                                             |
| `templates/plan-template.md`                                                                                | **Réutiliser tel quel**                                                          |
| `templates/tasks-template.md`                                                                               | **Réutiliser tel quel**                                                          |
| `templates/checklist-template.md`                                                                           | **Réutiliser tel quel**                                                          |
| `templates/constitution-template.md`                                                                        | **Réutiliser tel quel**                                                          |
| `templates/agent-file-template.md`                                                                          | **Réutiliser, compléter** (ajouter sections backlog/review)                      |
| `templates/commands/speckit.{specify,clarify,plan,tasks,analyze,implement,merge,constitution,checklist}.md` | **Réutiliser, patcher** pour retirer les étapes manuelles et câbler l'auto-chain |
| `scripts/bash/*.sh`                                                                                         | **Réutiliser tel quel** (déjà POSIX propre)                                      |
| `templates/commands/speckit.review.md`                                                                      | **Nouveau** (n'existe pas upstream)                                              |
| `commands/backlog.md`                                                                                       | **Nouveau**                                                                      |
| `agents/product-owner.md`, `review-coordinator.md`, `code-reviewer.md`, …                                   | **Nouveau** (dérivés des exemples, nettoyés)                                     |
| `skills/speckit/SKILL.md`                                                                                   | **Nouveau** (auto-chain dispatcher)                                              |

`spec-kit/LICENSE`). Les parties réutilisées conservent l'attribution ; `LICENSE` de Specflow est
aussi MIT et inclut la mention

## 6. Gestion des installs existantes (`--force`, `upgrade`)

**Cas 1 — dossier vierge** : aucun `.claude/` ni `.specify/` → écriture directe.

**Cas 2 — `.claude/` existe (l'utilisateur a déjà du Claude Code config)** :

- Sans `--force` → abandon avec erreur "present — use `specflow upgrade` or `--force`".
- Avec `--force` :
  - Fichiers que Specflow gère (liste déclarative dans `templates/manifest.json`) → overwrite.
  - Fichiers utilisateur inconnus → laissés intacts.
  - Conflits connus (ex. utilisateur a un `speckit.implement.md` custom) → sauvegarde en
    `.specflow.bak` et remplace.

**`upgrade`** : parcourt le manifest, compare checksums, liste les diffs, prompte pour chaque
changement (`overwrite / keep / show diff`). Version manifest = version templates.

## 7. Distribution

### Cibles `deno compile`

```
deno compile --target x86_64-unknown-linux-gnu  -o dist/specflow-linux-x64    src/main.ts
deno compile --target aarch64-unknown-linux-gnu -o dist/specflow-linux-arm64  src/main.ts
deno compile --target x86_64-apple-darwin       -o dist/specflow-macos-x64    src/main.ts
deno compile --target aarch64-apple-darwin      -o dist/specflow-macos-arm64  src/main.ts
deno compile --target x86_64-pc-windows-msvc    -o dist/specflow-windows-x64.exe src/main.ts
```

### Canaux

- **GitHub Releases** — tags `vX.Y.Z`, workflow release.yml déclenché sur tag. Upload des 5
  binaires + checksums SHA256.
- **Homebrew tap personnel** (`kevinr/homebrew-tap`) — formule Ruby qui pointe sur le release.
  Auto-update à chaque release via PR.
- **`curl | sh` installer** — script `install.sh` sur GitHub Pages ou dans le repo
  (`https://specflow.dev/install.sh` si domaine, sinon raw github URL). Détecte OS/arch, télécharge
  le bon binaire.

### Versioning

- Semver strict.
- Version du binaire + version des templates embarqués affichées par `--version`. Ex :
  `specflow 0.1.0 (templates 0.1.0)`. Elles peuvent diverger à partir de v0.2.

## 8. Gestion d'erreurs et edge cases

| Cas                                       | Comportement                                                                                           |
| ----------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| Pas de `git` dans le PATH                 | `init` continue (on ne bloque pas), `check` avertit. `backlog sync` refuse.                            |
| Pas de `gh` dans le PATH                  | `init` continue, `backlog sync` refuse avec instruction d'install.                                     |
| Network absent (install)                  | Les templates étant inlined dans le binaire, `init` fonctionne **hors-ligne**. Aucun download runtime. |
| `.claude/` existe déjà                    | Voir §6.                                                                                               |
| Répertoire cible n'existe pas             | Créé récursivement si `init <path>`.                                                                   |
| Nom de projet invalide (caractères shell) | Refusé avec message.                                                                                   |
| Terminal non-interactif (CI)              | Toutes les interactions sont par défaut "non" ; flags explicites requis.                               |
| `gh` non authentifié                      | `backlog sync` détecte via `gh auth status` et propose `gh auth login`.                                |

## 9. Tests

- **Unit** : parsing de manifest, rendu de templates, détection d'environnement. Cible ~70 % ligne
  coverage.
- **Integration** : `deno test` qui spawn `specflow init` dans un `Deno.makeTempDir()`, vérifie :
  - arborescence attendue,
  - contenu de fichiers clés (checksums),
  - idempotence (`init` sur un dossier vierge × 2),
  - `--force` overwrite correct.
- **Smoke CI** : matrice GH Actions sur les 3 OS (ubuntu, macos, windows) pour valider que
  `specflow init` produit bien les mêmes fichiers partout.

## 11. Risques et questions ouvertes

1. **Dérive des templates upstream** — Spec Kit upstream évolue (ils sont à v0.8.1.dev). Il faut un
   process de veille (diff périodique) pour ne pas accumuler trop de dette. Planifier un rebase
   trimestriel minimum.
2. **Licence** — confirmé MIT (GitHub, Inc.). Attribution explicite requise dans `LICENSE` et
   headers de fichiers réutilisés.
3. **Taille binaire** — `deno compile` produit ~80 MB. Supportable mais à annoncer clairement. Pas
   de solution sauf migrer vers Rust/Go plus tard.
4. **Templates embedding** — attendre la stabilisation de
   `import text from "./x.md" with { type: "text" }` en Deno, ou utiliser un script pré-compile qui
   génère un module `templates.ts` contenant tous les contenus. Option 2 retenue en v0.1 pour zéro
   dépendance aux features non-stables.
5. **Python dans le sync script** — résolu : on réécrit en TypeScript et on le met **dans le
   binaire** (sous-commande `specflow backlog sync`).
6. **Support Windows pour les scripts bash** — upstream a des `scripts/powershell/` pour Windows. À
   embarquer aussi ou à remplacer par des scripts Deno exécutés via `deno run` (= dépendance Deno
   côté user). Décision : **embarquer les deux** versions bash + powershell (comme upstream). Pas de
   dépendance Deno ajoutée côté user.
