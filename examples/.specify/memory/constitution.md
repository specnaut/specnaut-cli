# Miximodel Constitution

## Core Principles

### I. Clean Architecture (MVC + Service Layer)

Every feature MUST follow the layered architecture: **Controller → Service →
Repository**.

- **Controllers** (`app/controllers/`) are thin HTTP handlers. They MUST NOT
  contain business logic. Their sole responsibility is to validate input,
  delegate to a service, and return a response via `inertia.render()`.
- **Services** (`app/services/`) encapsulate all business logic. They MUST use
  `@inject()` for dependency injection and remain framework-agnostic where
  possible.
- **Repositories** (`app/repositories/`) are the only layer permitted to
  interact with the database via Lucid ORM. No raw SQL queries are allowed.
- No layer may skip another: a controller MUST NOT call a repository directly,
  and a service MUST NOT render views.

### II. Type Safety & Validation First

All data boundaries MUST be explicitly typed and validated.

- Every incoming HTTP request MUST be validated with a **VineJS** validator
  (`app/validators/`). No controller action may proceed without validation.
- All models MUST use Lucid ORM decorators (`@column`, `@hasMany`, etc.) with
  explicit TypeScript types.
- The auto-generated `database/schema.ts` MUST NEVER be edited manually — it is
  regenerated by `node ace migration:run`.
- Inertia page props MUST be typed via shared TypeScript interfaces to ensure
  end-to-end type safety between backend and frontend.

### III. Frontend Architecture & Design System

The UI MUST use the established design system consistently and adhere to strict feature-driven architecture.

- **Feature-Driven Architecture**: All business components, hooks, and domain-specific state MUST reside in `inertia/features/<domain>/`. The `inertia/components/` directory is STRICTLY reserved for generic, reusable UI elements without business logic.
- **Smart vs Dumb Components**: UI components must act as views (Dumb). They MUST NOT make direct network calls (e.g., using `fetch()` or Axios). Data fetching and navigation routing logic (`router`, `usePage`) must be lifted to Inertia pages (`inertia/pages/`) and passed down via props, or encapsulated in dedicated custom hooks.
- **Form Management**: All forms MUST exclusively use Inertia's `useForm` hook. Any input field subject to backend validation MUST be accompanied by the standard `<FieldError name="..." />` component to map VineJS errors natively.
- All UI components MUST be built with **shadcn/ui** primitives and **Tailwind CSS**. Custom CSS is prohibited.
- All components MUST use the project's CSS design token variables (e.g.,
  `--radius`, `--primary`, etc.) for consistent theming.
- ShadCN UI components retain their **kebab-case** naming convention (e.g.,
  `alert-dialog.tsx`).
- Navigation between pages MUST use the Inertia `Link` component for SPA
  behavior.

### IV. Convention Over Configuration

The project enforces strict naming and formatting conventions to ensure
consistency and reduce cognitive overhead.

- All filenames MUST use **snake_case** (e.g., `user_service.ts`,
  `not_found.tsx`), except shadcn/ui components which use kebab-case.
- **Prettier** with `@adonisjs/prettier-config` is the sole formatter: single
  quotes, no semicolons, ES5 trailing commas, 2-space indentation, 100-character
  print width, always arrow parens, bracket spacing.
- All code MUST pass Prettier formatting checks. Non-compliant code MUST NOT be
  committed.
- `@adonisjs/auth` is the only authentication library. Two guards are
  permitted:
  - **Session guard** — the default mechanism for browser-based flows (Inertia
    pages, user-facing forms, cookie-authenticated requests).
  - **Access token guard** — reserved for programmatic callers (admin tooling,
    CI, headless scripts, and future mobile clients). Tokens MUST be stored
    hashed at rest, be individually revocable, and MUST NOT be used to
    authenticate browser-rendered Inertia pages. Token-protected endpoints MUST
    live under an API-style path (e.g., `/api/*`) separate from Inertia page
    routes.
  - No other authentication library or hand-rolled credential scheme is
    permitted.

### V. Testability by Design

Every new service and critical path MUST be designed for testability.

- Services MUST use `@inject()` dependency injection so that dependencies can be
  mocked in unit tests.
- Business logic MUST reside in services (not controllers or models) to enable
  isolated unit testing.
- Migrations MUST include a reversible `down()` method to support rollback
  during development and testing.
- Tests use the **Japa** test framework with `unit`, `functional`, and `browser`
  test suites.

### VI. Simplicity & YAGNI

Start with the simplest solution that meets the requirement.

- Do not add abstractions, patterns, or dependencies that are not required by
  the current feature. Justify any complexity increase.
- Prefer Lucid ORM built-in features over custom implementations.
- Prefer Inertia.js conventions (server-driven navigation, shared data) over
  building a separate API layer.
- Every new dependency or pattern MUST be justified against the principle: "Does
  this solve a problem we have today?"

### VII. Client vs Server State (Zustand)

State management must clearly separate Server State from ephemeral Client State.

- **Server State**: Data that lives in the database (lists, collections, models) MUST be managed by Inertia.js (passed via props from controllers). NEVER duplicate Server State into a local store.
- **Client State**: Ephemeral UI properties, unsubmitted multi-step drafts, and real-time SSE (Adonis Transmit) events MUST use **Zustand**.
- Zustand stores must be domain-specific and located within their feature directory (e.g., `inertia/features/chat/store/use_chat_store.ts`).

### VIII. Infrastructure as Injectable Dependencies

All infrastructure services MUST be treated as injectable dependencies, not
static singletons.

- **Cache**, **message queues**, **external API clients**, **file storage**, and
  any I/O-bound service MUST be injected via constructor dependency injection —
  never imported as static module-level singletons.
- This ensures that unit tests can substitute a mock or in-memory implementation
  without relying on real infrastructure or managing global state between test
  suites.
- Infrastructure calls that bypass DI create hidden coupling: tests become
  order-dependent, cache entries leak between test suites, and services cannot be
  tested in isolation.
- When a service needs caching, it receives a cache interface via its
  constructor. The service does not know or care whether the underlying
  implementation is in-memory, Redis, or a no-op stub.
- **Graceful degradation**: Any infrastructure dependency MAY fail at runtime
  (network errors, timeouts). Services MUST handle failures gracefully — fall
  back to direct computation rather than propagating errors to end users. Cache
  is a performance optimization, never a correctness requirement.

### IX. Event-Driven Side Effects

All side effects triggered by user actions MUST be decoupled from business
services via the AdonisJS Emitter.

- **Side effects** include: notifications, automated chat messages, background
  processing, external API sync, activity logging, and any operation that is not
  part of the primary business transaction.
- Business services MUST emit typed domain events (extending `BaseEvent`) and
  MUST NOT import or call `NotificationService` directly. Only event listeners
  and the notifications controller may import `NotificationService`.
- Event listeners MUST wrap their bodies in try/catch — a listener failure MUST
  NOT propagate to the calling service or affect the primary business operation.
- Each event class MUST be placed in `app/events/<domain>/` and registered in
  `start/events.ts` with the appropriate listener.
- The `NotificationDispatcher` listener (`app/listeners/notification_dispatcher.ts`)
  is the single gateway for all notification creation. It checks user notification
  preferences before dispatching.

## Technology Constraints

The following technology choices are non-negotiable and MUST NOT be changed
without a constitutional amendment.

| Layer        | Technology                             | Version |
| :----------- | :------------------------------------- | :------ |
| Backend      | AdonisJS with TypeScript               | v7      |
| Frontend     | Inertia.js + React                     | Latest  |
| Client State | Zustand                                | Latest  |
| UI Library   | shadcn/ui + Tailwind CSS               | Latest  |
| Database     | PostgreSQL via Lucid ORM               | —       |
| Auth         | Session-based (`@adonisjs/auth`)       | —       |
| Validation   | VineJS                                 | —       |
| Testing      | Japa                                   | —       |
| Formatting   | Prettier (`@adonisjs/prettier-config`) | —       |

Adding a new core dependency (ORM, framework, auth provider) requires a
constitutional amendment with migration plan.

## Development Workflow

### Feature Implementation Flow

1. **Validate** — Ensure a VineJS validator exists for the feature's input
   before writing any controller code.
2. **Migrate** — Create the database migration (with `down()` method), run it,
   and let `database/schema.ts` auto-regenerate.
3. **Model** — Create or update Lucid models extending the generated schema
   classes.
4. **Repository** — Implement data access in a repository class.
5. **Service** — Implement business logic in a service with `@inject()`.
6. **Controller** — Wire the thin controller to the service.
7. **Page** — Build the Inertia React page and components.
8. **Test** — Write unit tests for the service and functional tests for the
   controller endpoints.

### Code Review Standards

- Every change MUST pass Prettier formatting.
- Every migration MUST include a reversible `down()` method.
- Controllers MUST NOT contain business logic.
- No raw SQL — all queries through Lucid ORM.

## Governance

- This constitution supersedes all other practices. When in conflict with
  informal conventions, the constitution wins.
- Amendments require:
  1. A documented rationale explaining the change.
  2. An impact assessment on existing code.
  3. A migration plan if the amendment affects existing patterns.
  4. Version bump following semantic versioning (MAJOR for principle
     removals/redefinitions, MINOR for additions, PATCH for clarifications).
- All code reviews MUST verify compliance with these principles.
- The `AGENTS.md` file at the project root serves as the quick-reference runtime
  guidance and MUST stay aligned with this constitution.
