# Feature Specification: Reliable Adoption guide in CI-generated release notes

**Feature Branch**: `010-release-adoption-parity` **Created**: 2026-06-09 **Status**: Draft
**Input**: GitHub issue mkrlabs/specflow#363 — "fix(release): adoption guide section missing from
CI-generated release body"

## User Scenarios & Testing _(mandatory)_

The "users" of a Specflow release body are (a) the **downstream AI agent** that reads a release to
learn how to adopt new features in an existing project, and (b) the **release engineer** who
publishes the release and trusts the local preview to match what ships. The bug breaks both: the
locally-previewed Adoption guide is silently absent from the published release.

### User Story 1 - CI release body carries the Adoption guide (Priority: P1)

When a release is published from the CI pipeline, its body contains the `### Adoption guide` section
for every feature whose pull request included a valid adoption block — exactly as the local release
preview showed. The downstream agent reading the published release gets the per-feature adoption
prompts; nothing the engineer verified locally goes missing on publish.

**Why this priority**: This is the headline regression. Every CI-published release since the defect
is missing the primary affordance that makes a Specflow release consumable by an agent. Restoring it
is the whole point of the issue; it is independently shippable and demonstrable on its own.

**Independent Test**: Take a commit range containing at least one `feat` PR that carried a valid
adoption block. Generate the release body the same way the publish pipeline does, with the same
authentication context the pipeline provides. Confirm the resulting body contains the
`### Adoption guide` section with that feature's entry.

**Acceptance Scenarios**:

1. **Given** a release range with N feature PRs that each carry a valid adoption block, **When** the
   body is generated by the publish pipeline, **Then** the published body contains an
   `### Adoption guide` section with N entries.
2. **Given** the same range and the same set of PR contents, **When** the body is generated locally
   and by the pipeline, **Then** the `### Adoption guide` section is byte-identical between the two.
3. **Given** the v1.13.0 release range (which contains feature PRs with adoption blocks), **When**
   the body is regenerated by the pipeline path, **Then** the Adoption guide is present (the
   published v1.13.0 body had it missing).

---

### User Story 2 - Parity is protected by an automated guard (Priority: P1)

The pipeline can no longer silently publish a body that is missing an Adoption guide the local path
would have produced. An automated check detects a parity break and stops the release (or fails CI)
before an incomplete body reaches users.

**Why this priority**: Without a guard the regression can silently return the moment the
authentication context, checkout depth, or any upstream dependency shifts again — exactly how it was
introduced unnoticed. The fix is only durable if a check makes the parity guarantee hard to lose. It
is independently testable against a deliberately-broken generation context.

**Independent Test**: Force the generation context into the broken state (the state that produces a
missing guide). Confirm the automated guard reports a failure / non-zero exit instead of yielding an
incomplete body that publishes successfully.

**Acceptance Scenarios**:

1. **Given** a generation context that cannot produce an Adoption guide that the local path would
   produce, **When** the release pipeline runs, **Then** the pipeline fails (non-zero / blocked
   publish) with a message naming the missing adoption entries.
2. **Given** a healthy generation context, **When** the guard runs, **Then** it passes without
   blocking the release.
3. **Given** the guard, **When** it runs, **Then** it executes automatically on every release (and
   is runnable on demand for a given range).

---

### User Story 3 - Genuine omissions stay quiet; infrastructure failures are loud (Priority: P2)

A feature PR that legitimately carries no adoption block is correctly and quietly skipped (no guide
entry, no error). A failure to _retrieve_ a PR's contents — authentication, rate-limiting, network —
is treated as a defect, not as "this PR has no adoption block." The two causes of "no entry" are
never conflated.

**Why this priority**: The root failure mode is that an infrastructure error and a
legitimately-absent section both currently resolve to the same silent "skip." Distinguishing them is
what lets the US2 guard fire on real problems without false positives on PRs that were never meant
to contribute a prompt. P2 because US1+US2 already restore and protect the visible behaviour; this
hardens the distinction so the guard is trustworthy.

**Independent Test**: Generate a body against (a) a feat PR with no adoption block, and (b) a feat
PR whose contents cannot be retrieved. Confirm (a) produces a quiet, logged skip and a successful
run, while (b) is reported as a retrieval failure rather than a silent skip.

**Acceptance Scenarios**:

1. **Given** a feature PR with no adoption block, **When** the body is generated, **Then** it is
   skipped with an informational log and the run succeeds.
2. **Given** a feature PR whose contents cannot be retrieved (auth / rate-limit / network), **When**
   the body is generated, **Then** the failure is surfaced distinctly from a legitimate absence and
   does not silently drop the entry.

### Edge Cases

- **Partial retrieval failure** — some feature PRs are retrieved and some fail mid-run (e.g.
  rate-limit hit after the first few). The body must not publish with a partial Adoption guide
  presented as complete.
- **No feature PRs in range** — a release with only fixes/chores legitimately has no Adoption guide
  section; this is success, not a parity break.
- **Feature commit without a PR reference** — a `feat` subject with no trailing PR number cannot be
  associated with adoption content; behaviour must be defined (skip vs. surface) and consistent
  between local and CI.
- **Feature PR with an adoption section but no prompt content** — already treated as "incomplete"
  and excluded; this must remain a legitimate quiet skip, not a guard failure.
- **Re-run of the same release** — regenerating the body for an already-tagged release produces the
  same Adoption guide (idempotent), so a re-run after fixing a transient failure recovers the
  complete body.

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: The release-notes generation MUST include an `### Adoption guide` entry for every
  in-range feature PR that carries a valid adoption block, regardless of whether generation runs
  locally or in the publish pipeline.
- **FR-002**: For an identical commit range and identical PR contents, the Adoption guide produced
  by the publish pipeline MUST be identical to the one produced locally.
- **FR-003**: The publish pipeline MUST provide the release-notes generation with whatever access it
  needs to retrieve PR contents, so that retrieval succeeds in CI under the same conditions it
  succeeds locally.
- **FR-004**: A failure to retrieve a PR's contents MUST be distinguishable from a PR that
  legitimately has no adoption block; the former MUST NOT be silently treated as the latter.
- **FR-005**: The system MUST NOT publish a release body that is missing an Adoption guide entry the
  local path would have produced; such a condition MUST stop the release or fail the pipeline.
- **FR-006**: An automated parity check MUST exist that fails (non-zero exit / blocked publish) when
  the pipeline-generated Adoption guide diverges from the expected one for the same range, and MUST
  run automatically on every release.
- **FR-007**: A feature PR that legitimately carries no adoption block MUST continue to be skipped
  quietly (informational log only) and MUST NOT cause a parity-check failure.
- **FR-008**: The fix MUST identify and remediate the root cause of the divergence (not only mask
  the symptom); the remediation MUST hold across the conditions under which the pipeline runs
  (authentication context, checkout history depth, and PR-content retrieval).
- **FR-009**: The release engineer MUST be able to reproduce, for a given range, the exact Adoption
  guide the pipeline will publish, so the local preview is a faithful predictor of the published
  body.

### Key Entities _(see Domain Model section below for full structure)_

- **Release body** — the published Markdown describing a release, including the Adoption guide
  section.
- **Adoption entry** — one feature's adoption prompt, derived from that feature's PR, that appears
  in the Adoption guide.
- **Generation context** — the environment in which the body is produced (local vs. pipeline),
  including its access to PR contents and commit history.

## Domain Model _(mandatory)_

**Bounded context:** Release changelog generation.

**Vocabulary (Ubiquitous language):**

- **Release body** — the Markdown document published as the GitHub Release description for a tag.
- **Adoption guide** — the `### Adoption guide` section of the release body; the collection of
  per-feature adoption prompts for that release.
- **Adoption entry** — one feature's prompt block within the Adoption guide, sourced from that
  feature's pull-request description.
- **Adoption block** — the `## Agent adoption` section in a PR description that supplies an adoption
  entry; "valid" iff it contains prompt content.
- **Generation context** — the runtime conditions under which the body is produced: local engineer
  machine vs. publish pipeline, and the access to PR contents and commit history each provides.
- **Parity** — the property that the same range and PR contents yield the same Adoption guide in
  every generation context.
- **Parity guard** — the automated check that fails the release when parity is violated.

**Entities (have identity):**

- **Release** [aggregate root] — identified by its version tag; owns the release body and the set of
  adoption entries derived from its range.
- **Feature pull request** — identified by its PR number; the source of at most one adoption entry.

**Value objects (no identity, immutable):**

- **AdoptionEntry(prNumber, title, prompt)** — one feature's contribution to the guide; exists iff
  its PR carries a valid adoption block.
- **GenerationContext(retrievalAccess, historyDepth)** — the access conditions of a run; parity
  requires the pipeline context to be sufficient for the same retrieval the local context performs.
- **RetrievalOutcome** — one of {retrieved, legitimately-absent, retrieval-failed}; the invariant
  below forbids collapsing the third into the second.

**Invariants (rules the domain must never break):**

- A feature PR with a valid adoption block that is in range MUST yield an adoption entry in every
  generation context — environment must not change the output.
- A release body MUST NOT be published while missing an adoption entry that the local path would
  have produced (no silent incompleteness).
- "No adoption entry for a PR" MUST be attributable to a defined cause — legitimately-absent or
  surfaced retrieval failure — never to an unobserved silent failure.

**Out of scope (other bounded contexts touched but not owned here):**

- **PR-adoption authoring / linting** — the `## Agent adoption` block's required structure and the
  PR-time lint that enforces it are owned elsewhere; this feature consumes the block, it does not
  define or validate its authoring rules.
- **Security preflight** — the release workflow's security gate is unrelated; it only shares the
  workflow file.
- **Release publishing / distribution** — building binaries, tap/marketplace syncs, and uploading
  assets are unchanged; only the body content is in scope.

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: For a release range containing N feature PRs with valid adoption blocks, the
  pipeline-published body contains exactly N adoption entries — the same count the local preview
  produces for that range.
- **SC-002**: For identical range and PR contents, the Adoption guide section is byte-identical
  between the local and pipeline generations.
- **SC-003**: Zero releases publish with an Adoption guide that is missing an entry the local path
  would have produced; any such condition halts the release instead.
- **SC-004**: Regenerating the body for the v1.13.0 range reproduces the defect before the fix
  (Adoption guide absent) and yields the complete guide after — the regression is demonstrably
  closed.
- **SC-005**: An automated parity check runs on every release and exits non-zero when the pipeline
  Adoption guide diverges from the expected one for the range.
- **SC-006**: A feature PR that legitimately lacks an adoption block produces no parity-check
  failure (zero false positives from legitimate absences).

## Assumptions

- The defect is environmental: the same generation code produces the guide locally but not in the
  pipeline, so the root cause lies in the conditions the pipeline provides to that code (PR-content
  retrieval access and/or commit history), not in the adoption-extraction logic itself. The plan
  phase confirms the specific cause.
- Amending already-published release bodies (e.g. v1.13.0) is **out of scope** per
  `memory/never_amend_published_releases.md`; the fix targets the next release (v1.13.1+).
- The `## Agent adoption` block structure and the PR-time adoption lint are unchanged; this feature
  relies on them as-is.
- The local release preview path and the pipeline path are expected to share the same generation
  logic; the fix preserves that single shared path rather than forking behaviour per environment.
- A representative range with known adoption-bearing feature PRs is available for regression testing
  (the v1.13.0 range is one such fixture).
